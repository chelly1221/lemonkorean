# 개발노트 자동 생성을 위한 Claude 작업 프로토콜

## 개요
CLAUDE.md에 중요한 코드 변경을 완료한 후 Claude가 자동으로 개발노트를 생성하도록 명시적으로 지시하는 새로운 "Claude 작업 프로토콜 (Claude Work Protocol)" 섹션을 추가했습니다.

## 문제 / 배경
이전에는 CLAUDE.md에 개발노트를 **어떻게** 작성하는지(형식, 구조, 예시)를 설명하는 포괄적인 가이드가 포함되어 있었지만, Claude에게 표준 워크플로우의 일부로 개발노트를 능동적으로 생성하라는 명확한 **명령적 지시**가 부족했습니다.

기존 "개발노트 작성 가이드"는 설명적이었지 규범적이지 않았습니다:
- ✅ 노트를 **작성해야 하는** 시기를 설명함
- ✅ 템플릿과 예시를 제공함
- ✅ 모범 사례를 보여줌
- ❌ Claude에게 노트를 자동으로 생성하라고 명시적으로 지시하지 **않음**
- ❌ 명확한 워크플로우 통합 없음

이는 개발노트 생성 여부가 다음에 달려 있음을 의미했습니다:
1. Claude가 설명적 가이드를 해석하는 방식
2. 사용자가 명시적으로 개발노트를 요청하는 경우
3. 세션 간 일관성 없는 준수

## 해결 방법 / 구현
다음을 제공하는 눈에 띄는 새 섹션 "Claude 작업 프로토콜"을 추가했습니다:

### 1. 명확한 명령적 지시
**"Claude는 중요한 코드 변경을 완료한 후 반드시 개발노트를 작성해야 합니다"**

이는 정보 제공식(어떻게 하는가)이 아닌 행동 지향식(반드시 해야 함)입니다.

### 2. 구조화된 워크플로우
명확한 트리거와 우선순위가 있는 5가지 카테고리:
- 새로운 기능 구현 (medium ~ high)
- 버그 수정 (심각도에 따라 high/medium/low)
- 아키텍처 변경 (high)
- 성능 최적화 (medium)
- 문서 업데이트 (low ~ medium)

### 3. 단계별 프로세스
1. 작업 완료 확인
2. 개발노트 파일 생성 (명명 규칙)
3. Frontmatter 작성 (카테고리/우선순위 포함)
4. 콘텐츠 작성 (필수 섹션)
5. 파일 경로와 함께 사용자에게 알림

### 4. 명시적 예외
노트를 생성하지 **않아도** 되는 경우의 명확한 목록:
- 단순 타이포 수정 (1-2줄)
- 로그 메시지 변경
- 코드 포맷팅만
- 주석 추가
- 설정 미세 조정

### 5. 분류 가이드
- 카테고리 가이드 (Mobile, Backend, Frontend 등)
- 우선순위 가이드 (high, medium, low 기준)
- 완전한 작업 예시

### 6. 전략적 배치
"프로젝트 개요" 및 "파일 통계" 이후, "아키텍처" 이전에 삽입:
- 조기 가시성 (Claude가 즉시 읽음)
- 눈에 띄는 위치 (문서에 묻히지 않음)
- 행동 우선 접근법

## 관련 이슈 / 참고사항

### 보완 섹션
이 새로운 프로토콜 섹션은 기존 상세 가이드를 **대체하지 않고 보완**합니다:
- **프로토콜 섹션** (신규): 행동 지향적, 명령적, "반드시 해야 함"
- **상세 가이드** (기존): 참조 지향적, 설명적, "어떻게 하는가"

함께 다음을 제공합니다:
- Claude의 동작에 대한 명확한 지시
- 참조를 위한 상세한 템플릿과 예시
- 완전한 워크플로우 통합

### Claude의 워크플로우에 미치는 영향
이 변경은 Claude의 동작을 표준화합니다:
- **이전**: 일관성 없는 개발노트 생성
- **이후**: 중요한 변경에 대한 자동 개발노트 생성
- **이점**: 더 나은 프로젝트 문서화 및 변경 추적

### 향후 고려사항
- 개발노트 생성 빈도를 모니터링하여 적절한 빈도 보장
- 너무 많거나 적은 노트가 생성되면 우선순위 임계값 조정
- 개발노트를 git 커밋과 연결하는 자동화 고려
- 변경 로그 생성과의 통합 가능성

### 메타 노트
이 개발노트 자체는 방금 추가한 프로토콜에 따라 생성되었으며, 워크플로우가 실제로 작동함을 보여줍니다. 프로토콜은 자기 강화적입니다 - 이를 따름으로써 효과를 검증하고 있습니다.

## 배운 교훈

1. **명시적이 더 낫다**: 설명적 가이드만으로는 충분하지 않음; Claude에게 명시적 지시가 필요함
2. **배치가 중요**: 조기에 눈에 띄는 섹션이 묻힌 문서보다 더 많은 주목을 받음
3. **이론보다 프로세스**: 단계별 워크플로우가 개념적 설명보다 실행 가능함
4. **예외의 중요성**: 무엇을 하지 말아야 하는지 아는 것이 무엇을 해야 하는지 아는 것만큼 중요함
5. **예시가 채택을 촉진**: 완전한 작업 예시가 프로토콜을 즉시 사용 가능하게 만듦
